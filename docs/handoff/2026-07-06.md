## What happened this session

Built Step 8 — Filter bar — test-first (red → green), scoped down mid-session:

1. **FilterBar component** (`components/Board/FilterBar.tsx` + `.test.tsx`): a single-select `<select>` for label filtering, with a "Clear filters" button shown only when a filter is active. Rendered inside `BoardCanvas` (above the lists, below `BoardHeader`) rather than as a separate component in `app/board/[id]/page.tsx`, since filter state is client-only and `BoardCanvas` already owns `cardsByList`/label data.

2. **Filtering wired into `BoardCanvas`**: non-matching cards are hidden via a `hidden` prop threaded to `SortableCardTile` → applies a `hidden` class to the card's wrapper `div` (marked `data-card-wrapper` for testability) rather than removing the card from `cardsByList`. This keeps `SortableContext` indices and drag-and-drop position math untouched.

3. **Priority filter descoped**: the original scope (from `project-manager-plan.md`) was "filter by label and priority," and `cards.priority` already exists in the schema. But there's currently **no UI anywhere to set a card's priority** — `CardModal` only has title and labels. Filtering by an unsettable field would be dead functionality, so priority filtering was cut from this session's work at the user's direction. `CardRow`'s optional `priority` field (added mid-session for this) was reverted since it wasn't needed once priority was descoped. **This is a real gap**: if Step 8 is revisited to add priority filtering, a priority selector needs to be added to `CardModal` first.

4. **A real regression was caught and fixed before it broke CI**: adding the label `<select>` meant label names now also appear as `<option>` text in the DOM, which collided with an existing ambiguous locator in `e2e/labels.spec.ts` (`page.getByText(LABEL_NAME)` matched both the new `<option>` and the label pill/button). Fixed by switching that locator to `page.getByRole("button", { name: LABEL_NAME })`, which the `<option>` doesn't match.

5. **A second real regression, same root cause class**: the new `e2e/filter-bar.spec.ts` creates a global label ("Filter QA") for its test data. Labels are global and are *not* deleted when a board is deleted, so without cleanup this label would leak into `e2e/labels.spec.ts`'s runs and break its assumption that exactly one label exists (manifested as "strict mode violation: 2 elements matched" on a `getByRole("button", { name: /rename/i })` locator, since every label row has a "Rename label" button). Fixed by adding an explicit "delete this label via the UI" step to `filter-bar.spec.ts`'s `afterAll`, before the board itself is deleted.

## Testing approach established (still applies)

For any new feature, split tests as:
- **Jest** for component/logic wiring — mock out third-party primitives that don't work well in jsdom (`@dnd-kit/core`, `@dnd-kit/sortable`, `@/components/ui/dropdown-menu`) and mock server actions (`@/lib/actions/*`) and `next/navigation`'s `useRouter`. See `components/Board/BoardCanvas.test.tsx` and `components/Label/LabelPicker.test.tsx` for the established mocking patterns.
- **Playwright** for real end-to-end flows and persistence-after-reload checks.
- Always write the test first, run it to confirm red, then implement.
- **New lesson this session**: when a Playwright locator is ambiguous only under certain global state (e.g. `getByText(name)` matching both a UI element and a newly-added `<option>`), prefer scoping by ARIA role (`getByRole("button", { name })`) over trying to scope by container — portal-rendered content (menus/popovers) often isn't nested under the container you'd expect (e.g. not under `role="dialog"`).
- **New lesson this session**: any E2E spec that creates a **global** entity (labels, and presumably anything similar in the future) must clean that entity up itself in `afterAll` — deleting the board it was created under is not sufficient, since global entities outlive the board.

## Environment notes

- `npx playwright test` starts its own dev server on port 3001 (see `playwright.config.ts`). If the user has their own `npm run dev` running (any port), Next's lock file blocks Playwright's server from starting — ask the user to stop their manual dev server before running the E2E suite, don't kill it yourself.
- Test DB lives at `data/test.db` (WAL mode) — safe to delete (`data/test.db*`) between runs if state gets weird; the real app DB is `data/project-manager.db`, never touch that. This session needed a delete-and-rerun after a failed test run left duplicate global labels behind.
- `workers: 1` is set intentionally in `playwright.config.ts` to avoid parallel writes to the shared SQLite test DB.
