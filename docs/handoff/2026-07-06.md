## Step 9's final model (superseding an earlier version built the same day)

Archive was first built as: active board/card menus show only "Archive"; archived board/card menus show only "Delete" (hard delete only reachable once archived); a per-list "Show archived (n)" toggle revealed archived cards inline within their original list via a flag (`cards.archived`).

**That per-list toggle was then replaced this session** with a different model, at the user's request:

- Archiving a card **physically moves it** into a board-wide "Archived" list (auto-created the first time it's needed, via `getOrCreateSpecialList` in `lib/actions/lists.ts`, shared by both card-archiving and list-deletion).
- The Archived list is **protected**: `ListHeader` hides its "···" options menu entirely and disables click-to-rename when `list.special` is true (new `lists.special` boolean column). It **persists even when empty** — nothing ever auto-deletes it.
- **Deleting a list now archives all its cards first** (into the Archived list), then deletes the (now-empty) list — previously this was a hard delete of the cards too, via cascading FK.
- **Restoring a card** (`restoreCard` in `lib/actions/cards.ts`) returns it to its original list if that list still exists on the board; if the original list was deleted in the meantime, it goes into a **single per-board "Restored" list** instead (also `special`, also auto-created/reused via the same helper — uniqueness enforced by looking up an existing `special` list with that title before creating one).
- New `cards.originalListId` column (plain integer, deliberately **not** a foreign key) records which list a card came from when archived, so `restoreCard` can check "does this list still exist" without the column itself being affected by that list's deletion.
- `CardModal`'s footer is now gated on the card's `archived` flag (still set/cleared by archive/restore, in addition to the physical list move): active card → "Archive" only; archived card (sitting in the Archived list) → "Restore" and "Delete" (permanent) side by side.
- Whole-board archiving (`archiveBoard`) was **deliberately left unchanged** — it still flags all the board's cards `archived = true` in place rather than moving them, since the board itself becomes inaccessible anyway. This was a conscious choice, not an oversight — confirmed with the user before implementing.
- The old `components/List/ArchivedCardsSection.tsx` (+ test) is deleted. `app/board/[id]/page.tsx` no longer splits cards into two queries (active vs. archived) — it fetches all of a board's cards in one query and lets each card's current `listId` (including the Archived/Restored lists, which are just regular rows in the `lists` table) determine where it renders. This actually simplified the plumbing versus the toggle-based version.

### Schema migration

`drizzle/0001_silly_catseye.sql` (generated via `npm run db:generate`): adds `lists.special` (boolean, default false) and `cards.original_list_id` (nullable integer, no FK). ## Real bugs/regressions caught this session

1. **Cross-cutting regression from Step 9's first pass** (two-step archive-then-delete for boards): every existing E2E spec cleaned up its test boards via a direct "Delete Board" click on an active board, which is no longer possible. Fixed by centralizing cleanup in a new `e2e/helpers.ts::deleteBoardViaUI(page, boardId)` and rewriting `boards-and-lists.spec.ts`'s delete tests into "Board archiving and deletion" tests that actually exercise the two-step flow.
2. **A real bug in that rewritten test itself**: it derived `boardId` from `page.url()` deep inside the test body (fragile), which flaked under full-suite ordering. Fixed by capturing `boardId` in the describe-level `beforeEach` immediately after board creation.
3. **User caught an unnecessary manual `rm -f data/test.db*`** before every Playwright run — `playwright.config.ts`'s `globalSetup` (`e2e/global-setup.ts`) already does this automatically. Dropped from the workflow.
4. **This session's second pass**: replacing the per-list toggle broke `e2e/cards.spec.ts`'s `afterAll` — its old cleanup logic tried to click a now-nonexistent "Archive" button on a card that had *already* been archived (and thus moved into the Archived list, where its modal now shows "Restore"/"Delete" instead), leaving a modal open and hanging the next locator for 30s. Fixed by deleting that cleanup logic entirely — it was already redundant, since hard-deleting the board at the end of the test cascades to every list and card regardless of archive state.

## Testing approach established (still applies)

- **Jest** for component/logic wiring — mock third-party primitives that don't work well in jsdom (`@dnd-kit/core`, `@dnd-kit/sortable`, `@/components/ui/dropdown-menu`) and mock server actions (`@/lib/actions/*`) and `next/navigation`'s `useRouter`. See `components/Board/BoardHeader.test.tsx`, `components/CardModal/index.test.tsx`, `components/List/ListHeader.test.tsx`, `components/Sidebar/ArchivedBoardsSection.test.tsx` for patterns — several of these components had **no test file before this feature**; started from scratch, test-first, each time.
- **Playwright** for real end-to-end flows, DB-touching server-action behavior (this repo doesn't unit-test `lib/actions/*` directly), and persistence-after-reload checks.
- Always write the test first, run it to confirm red, then implement.
- **Lesson**: a UI change that gates or removes an existing action has a blast radius across every E2E spec that used that action for its own test-data cleanup — grep all of `e2e/*.spec.ts` for the affected locator text before considering the change complete.
- **Lesson**: don't derive IDs or other test-critical values from `page.url()` deep inside a test body if navigation happened earlier in the same test — capture them immediately after the action that produces them.
- **Lesson**: a Playwright locator that's ambiguous only under certain global state (e.g. matching both a UI element and a newly-added `<option>`) is usually best fixed by scoping on ARIA role, not by trying to scope by container — portal-rendered content (menus/popovers) often isn't nested under the container you'd expect.
- **Lesson**: any E2E spec that creates a **global** entity (labels, or now Archived/Restored lists which are shared per-board) must think about what outlives the test — don't assume tearing down one thing cascades to everything a test created; but conversely, don't build fragile "clean up conditionally" logic when a coarser cleanup (e.g. hard-deleting the whole board) already makes it redundant.

## Environment notes

- `npx playwright test` starts its own dev server on port 3001 (see `playwright.config.ts`). If the user has their own `npm run dev` running (any port), Next's lock file blocks Playwright's server from starting — ask the user to stop their manual dev server before running the E2E suite, don't kill it yourself.
- **Don't manually `rm -f data/test.db*` before running Playwright** — `e2e/global-setup.ts` (wired via `globalSetup` in `playwright.config.ts`) already does this before every run.
- The real app DB is `data/project-manager.db` — never touch that.
- `workers: 1` is set intentionally in `playwright.config.ts` to avoid parallel writes to the shared SQLite test DB.
- Schema changes go through `npm run db:generate` (drizzle-kit) to produce a migration in `drizzle/`; it's applied automatically by `migrate()` in `lib/db.ts` the next time any process (dev server, Playwright) touches the DB.
