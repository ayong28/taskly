## Step 12's design system, in brief

`docs/design-system.md` is the source of truth — read it before touching styling again. Key decisions:

- **`:root` and `.dark` were merged onto one dark-only palette.** The brand brief mandates a dark background ("always display on a dark background") — there's no light variant of this brand, so keeping a separate light theme made no sense. The app has no theme toggle; if one gets added later it should be dark-only variations, not a light theme.
- **Full-saturation neon (`#00F0FF` cyan / `#FF00E5` magenta) is never used as a solid fill** — it fails contrast at UI sizes. Solid fills (buttons, active states) use the brief's own "muted" cyan/magenta (`#00C4D4` / `#D400C0`); full neon is reserved for borders, focus rings, and gradients/glows.
- **Fonts**: Orbitron (`--font-heading`) for headings/titles, Rajdhani (`--font-sans`) for body/UI text, JetBrains Mono (`--font-mono`) reserved for future code/terminal-flavored UI — loaded via `next/font/google` in `app/layout.tsx`.
- **No scanlines/starfield/grid-overlay backgrounds, no default per-element glow** — the brief itself marks these "optional"/"used sparingly," and they don't suit a dense, always-on-screen card list. Glow is reserved for specific interaction states (card hover, sidebar active link), not resting UI.
- Label colors (`lib/labels.ts`) were deliberately **left alone** — they're user-chosen category colors for visual differentiation, not decorative brand chrome, so swapping them for the 5 brand accent colors would defeat their purpose.

## A real regression caught mid-session (not part of Step 15 itself)

Running the full Playwright suite after the redesign (something not done after Steps 13/14 landed — only Jest/tsc were checked then) surfaced **6 failing specs**, all pre-existing breakage from Step 13: `archive.spec.ts`, `archive-list-deletion.spec.ts`, `cards.spec.ts`, `drag-and-drop.spec.ts`, `filter-bar.spec.ts`, `labels.spec.ts` all still drove card creation through the old inline `getByPlaceholder(/card title/i)` input, which `NewCardModal` had replaced with a labeled `Title` field inside a dialog. Fixed by pointing them at `page.getByRole("dialog").getByLabel(/title/i)` + the dialog's "Add card" button.

That fix then surfaced a **second, genuine race**: tests that create two cards back-to-back (e.g. `archive.spec.ts`'s `addCard` helper called twice) would sometimes fail on the second card — clicking the next list's "Add Card" trigger while the previous dialog was still mid-close-animation (100ms `data-closed:animate-out`) could get swallowed by the closing overlay. Fixed by asserting `await expect(page.getByRole("dialog")).not.toBeVisible()` after every "Add card" submit, before moving on. **Same lesson as the archive/restore race documented in the 2026-07-06 handoff: don't chain two dialog-driven actions without asserting the first one's dialog is actually gone.**

**Lesson for future sessions**: after a component-level UI change (especially one replacing an interaction pattern like inline-input → modal), run the full Playwright suite before considering the change done — Jest + tsc passing is not sufficient signal, since Jest mocks the very actions/components whose *wiring* changed.
